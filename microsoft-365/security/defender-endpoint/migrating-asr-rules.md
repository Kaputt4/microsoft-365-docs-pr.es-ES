# <a name="migrating-from-a-third-party-hips-to-asr-rules"></a>Migración de un HIPS de terceros a reglas ASR

Este artículo le ayuda a asignar reglas comunes a Microsoft Defender para endpoint. En la tabla siguiente se muestran preguntas y escenarios comunes al migrar de un producto DE HIPS de terceros a reglas ASR.

|Ámbito y acción|Procesos|Operación|Ejemplos de archivos/carpetas, claves/valores del Registro, procesos, servicios|Reglas de reducción de superficie de ataque|Otras características recomendadas|
|:--|:--|:--|:--|:--|:--|
|Todos los procesos: bloquear la creación de archivos específicos y claves del Registro||Creación de archivos|*.zepto, *.odin, *.locky, *.jaff, *.lukitus, *.wnry, *.krab|Las reglas ASR bloquean las técnicas de ataque y no los indicadores de compromiso (IOC). Bloquear una extensión de archivo específica no siempre es útil, ya que no impide que un dispositivo se vea comprometido. Solo frustra parcialmente un ataque hasta que los atacantes crean un nuevo tipo de extensión para la carga.|Se recomienda encarecidamente tener habilitado antivirus de Microsoft Defender, junto con la protección en la nube y el análisis de comportamiento. Se recomienda usar otra prevención, como la regla ASR "Usar protección avanzada contra ransomware". Esto proporciona un mayor nivel de protección contra ataques de ransomware. Además, Microsoft Defender para endpoint supervisa varias de estas claves del Registro, como las técnicas ASEP, que desencadenarán alertas específicas. Las claves del Registro usadas requieren un mínimo de privilegios de administrador local o instalador de confianza para poder modificarse. Se recomienda usar un entorno bloqueado, con derechos o cuentas administrativas mínimos. Se pueden habilitar otras configuraciones del sistema, como "Deshabilitar SeDebug para roles no obligatorios" que forman parte de nuestras recomendaciones de seguridad más amplias.|
|Todos los procesos: bloquear la creación de archivos específicos y claves del Registro||Modificaciones del Registro|*\Software \* ,HKCU\Environment\UserInitMprLogonScript,HKLM\SOFTWARE\Microsoft\Windows NT\CurrentVersion\Accessibility\ATs \* \StartExe, HKLM\SOFTWARE\Microsoft\Windows NT\CurrentVersion\Image File Execution Options \* \Debugger,HKEY_CURRENT_USER\Software\Microsoft\HtmlHelp Author\location,HKLM\SOFTWARE\Microsoft\Windows NT\CurrentVersion\SilentProcessExit \* \MonitorProcess|Las reglas ASR bloquean las técnicas de ataque y no los indicadores de compromiso (IOC). Bloquear una extensión de archivo específica no siempre es útil, ya que no impide que un dispositivo se vea comprometido. Solo frustra parcialmente un ataque hasta que los atacantes crean un nuevo tipo de extensión para la carga.|Se recomienda encarecidamente tener habilitado antivirus de Microsoft Defender, junto con la protección en la nube y el análisis de comportamiento. Se recomienda usar prevención adicional, como la regla ASR "Usar protección avanzada contra ransomware". Esto proporciona un mayor nivel de protección contra ataques de ransomware. Además, Microsoft Defender para endpoint supervisa varias de estas claves del Registro, como las técnicas ASEP, que desencadenarán alertas específicas. Además, las claves del Registro usadas requieren un mínimo de privilegios de administrador local o instalador de confianza para poder modificarse. Se recomienda usar un entorno bloqueado, con derechos o cuentas administrativas mínimos. Se pueden habilitar otras configuraciones del sistema, como "Deshabilitar SeDebug para roles no obligatorios" que forman parte de nuestras recomendaciones de seguridad más amplias.|
|Programas que no son de confianza desde USB: impedir que los programas que no son de confianza se ejecuten desde unidades extraíbles|*|Ejecución de procesos|*|Las reglas ASR tienen una regla integrada para impedir el inicio de programas que no son de confianza y que no están firmando unidades extraíbles: "Bloquear procesos que no son de confianza y sin firma que se ejecutan desde USB", GUID "b2b3f03d-6a65-4f7b-a9c7-1c7ef74a9ba4".|Explore controles adicionales para dispositivos USB y otros medios extraíbles con Microsoft Defender para endpoint: Cómo controlar dispositivos USB y otros medios extraíbles [con Microsoft Defender para Endpoint](https://docs.microsoft.com/windows/security/threat-protection/device-control/control-usb-devices-using-intune). |
|Mshta: impedir que Mshta inicie determinados procesos secundarios|mshta.exe|Ejecución de procesos|powershell.exe, cmd.exe, regsvr32.exe|Las reglas ASR no contienen ninguna regla específica para impedir que los procesos secundarios "mshta.exe". Este control está dentro de las competencias de Protección contra vulnerabilidades o Windows Defender control de aplicaciones.|Habilite Windows Defender control de aplicaciones para evitar mshta.exe que se ejecuten por completo. Si su organización requiere "mshta.exe" para aplicaciones de línea de negocio, configure una regla específica de protección contra vulnerabilidades de seguridad Windows Defender, para evitar que mshta.exe iniciar procesos secundarios.|
|Outlook: impedir que Outlook inicie procesos secundarios|outlook.exe|Ejecución de procesos|powershell.exe|Las reglas ASR tienen una regla integrada para impedir que las aplicaciones de comunicación de Office (Outlook, Skype y Teams) inicien procesos secundarios: "Bloquear la aplicación de comunicación de Office para crear procesos secundarios", GUID "26190899-1602-49e8-8b27-eb1d0a1ce869".|Se recomienda habilitar el modo de idioma restringido de PowerShell para minimizar la superficie de ataque de PowerShell.|
|Office: impedir que Las aplicaciones de Office inicien procesos secundarios y no creen contenido ejecutable|winword.exe, powerpnt.exe, excel.exe|Ejecución de procesos|powershell.exe, cmd.exe, wscript.exe, mshta.exe, EQNEDT32.EXE, regsrv32.exe|Las reglas ASR tienen una regla integrada para impedir que las aplicaciones de Office inicien procesos secundarios: "Bloquear todas las aplicaciones de Office para crear procesos secundarios", GUID "D4F940AB-401B-4EFC-AADC-AD5F3C50688A".|N/D|
|Office: impedir que Las aplicaciones de Office inicien procesos secundarios y no creen contenido ejecutable|winword.exe, powerpnt.exe, excel.exe|Creación de archivos|C:\Users \* \AppData \* *\* .exe, C:\ProgramData \** \* .exe, C:\ProgramData \* *\* .com, C:\Users \* \* AppData\Local\Temp* \* .com, C:\Users \* *\Downloads \** \* .exe, C:\Users \* \AppData \* *\* .scf, C:\ProgramData \** \* .scf, C:\Users\Public \* .exe, C:\Users \* \Desktop \* * \* .exe|N/D|
|Wscript: impedir que Wscript lea determinados tipos de archivos|wscript.exe|Lectura de archivo|C:\Users \* \AppData \* *\* .js*, C:\Users \* \Downloads \* *\* .js*|Debido a problemas de confiabilidad y rendimiento, las reglas de ASR no tienen la capacidad de impedir que un proceso específico lea un tipo de archivo de script determinado. Tenemos una regla para evitar vectores de ataque que puedan originarse en estos escenarios. El nombre de la regla es "Impedir que JavaScript o VBScript inicien contenido ejecutable descargado" (GUID "D3E037E1-3EB8-44C8-A917-57927947596D") y la "Ejecución de bloqueo de scripts potencialmente ofuscados" (GUID " 5BEB7EFE-FD9A-4556-801D-275E5FFC04CC")|Aunque hay reglas ASR específicas que mitigan ciertos vectores de ataque en estos escenarios, es importante mencionar que AV es capaz de inspeccionar scripts (PowerShell, Windows Script Host, JavaScript, VBScript y más) en tiempo real, a través de la Interfaz de examen antimalware (AMSI). Hay más información disponible aquí: Interfaz de [examen antimalware (AMSI).](https://docs.microsoft.com/windows/win32/amsi/antimalware-scan-interface-portal) |
|Adobe Acrobat: bloquear el inicio de procesos secundarios|AcroRd32.exe, Acrobat.exe|Ejecución de procesos|cmd.exe, powershell.exe, wscript.exe|Las reglas ASR permiten impedir que Adobe Reader inicie procesos secundarios. El nombre de regla es "Bloquear Adobe Reader de la creación de procesos secundarios", GUID "7674ba52-37eb-4a4f-a9a1-f0f9a1619a2c".|N/D|
|CertUtil: bloquear la descarga o creación de contenido ejecutable|certutil.exe|Creación de archivos|*.exe|Las reglas ASR no admiten estos escenarios porque forman parte de la protección antivirus de Microsoft Defender.|Microsoft Defender AV impide que CertUtil cree o descargue contenido ejecutable.|
|Todos los procesos: impedir que los procesos detengan componentes críticos del sistema|*|Terminación del proceso|MsSense.exe, MsMpEng.exe, NisSrv.exe, svchost.exe*, services.exe, csrss.exe, smss.exe, wininit.exe, etc.|Las reglas ASR no admiten estos escenarios porque están protegidas con protecciones de seguridad integradas de Windows 10.|ELAM (Early Launch AntiMalware), PPL (Protection Process Light), PPL AntiMalware Light y System Guard.|
|Procesos específicos: bloquear un intento de proceso de inicio específico|"Nombre del proceso"|Ejecución de procesos|tor.exe, bittorrent.exe, cmd.exe, powershell.exe y mucho más.|En general, las reglas ASR no están diseñadas para funcionar como administrador de aplicaciones.|Para evitar que los usuarios inicien programas o procesos específicos, la recomendación sería usar Windows Defender control de aplicaciones. Los indicadores de archivo de extremo y cert de Microsoft Defender se pueden usar en un escenario de respuesta a incidentes (no debe considerarse como un mecanismo de control de aplicaciones).|
|Todos los procesos: bloquear los cambios no autorizados en las configuraciones de MDATP AV|*|Modificaciones del Registro|HKLM\SOFTWARE\Policies\Microsoft\Windows Defender\DisableAntiSpyware, HKLM\SOFTWARE\Policies\Microsoft\Windows Defender\Policy Manager\AllowRealTimeMonitoring, etc.|Las reglas ASR no cubren este tipo de escenarios porque forman parte de la protección integrada de Microsoft Defender para endpoints.|La protección contra alteraciones (opt-in, administrada desde Intune) evita los cambios no autorizados en las claves del Registro DisableAntiVirus, DisableAntiSpyware, DisableRealtimeMonitoring, DisableOnAccessProtection, DisableBehaviorMonitoring y DisableIOAVProtection (y más). |
